# OctoAcme Project Management Documentation

Welcome to the OctoAcme Project Management documentation hub. This folder contains comprehensive process guides, templates, and best practices that help our teams deliver projects efficiently, consistently, and with confidence.

## Purpose

This documentation centralizes OctoAcme's project management approach, enabling:
- **Faster Onboarding**: New team members quickly understand our processes and expectations
- **Consistency**: All projects follow the same proven lifecycle and standards
- **Institutional Knowledge**: Tacit team insights are captured and searchable
- **Continuous Improvement**: Processes are versioned and improve over time based on retrospectives

## Core Principles

OctoAcme is built on these foundational principles:
- **Customer-first**: Prioritize customer value and usability in all decisions
- **Iterative delivery**: Deliver small, testable increments to reduce risk
- **Clear ownership**: Each project has a named Project Manager (PM) and Product Manager (PdM)
- **Data-informed decisions**: Measure impact and iterate based on evidence
- **Psychological safety**: Encourage feedback, learning, and blameless retrospectives

### Project Phases

The project lifecycle follows five key phases. Use these guides in order as your project progresses:

1. **[Project Initiation](octoacme-project-initiation.md)** — Validate business need, align stakeholders, and confirm go/no-go for planning. Outputs: One-pager, stakeholder list, initial risk list.

2. **[Project Planning](octoacme-project-planning.md)** — Break work into actionable increments, identify dependencies, and create a release plan. Outputs: Prioritized backlog, Definition of Done, milestone map.

3. **[Execution & Tracking](octoacme-execution-and-tracking.md)** — Manage day-to-day delivery, track progress, and handle blockers. Includes team rhythm, PR workflow, quality standards, and escalation paths.

4. **[Release & Deployment](octoacme-release-and-deployment.md)** — Standardize how features reach production. Covers release types, pre-release requirements, deployment checklist, and rollback procedures.

5. **[Retrospective & Continuous Improvement](octoacme-retrospective-and-continuous-improvement.md)** — Capture learnings after sprints or releases and convert them into actionable improvements.

### Cross-Cutting Concerns

- **[Risk Management & Communication](octoacme-risks-and-communication.md)** — Explains how to identify, assess, and communicate risks and dependencies. Includes risk register template, stakeholder communication cadence, and escalation paths.

## How to Use This Documentation

### I'm new to OctoAcme — where do I start?
1. Read the [Project Management Overview](octoacme-project-management-overview.md) to understand our high-level approach (5 min)
2. Review [Roles & Personas](octoacme-roles-and-personas.md) to find your role and responsibilities (5 min)
3. Bookmark this README and refer to phase-specific guides as your project progresses

### I'm starting a new project
1. Read [Project Initiation](octoacme-project-initiation.md) and complete the One-pager template
2. Get stakeholder buy-in using the examples provided
3. Move to [Project Planning](octoacme-project-planning.md) when approved

### I'm in active delivery — what do I need?
- Use [Execution & Tracking](octoacme-execution-and-tracking.md) for daily rhythms, PR workflow, and quality standards
- Reference [Risk Management & Communication](octoacme-risks-and-communication.md) for escalations and status updates
- Attend team syncs and demos as outlined in the Execution guide

### We're preparing to release
Follow [Release & Deployment](octoacme-release-and-deployment.md) to ensure a smooth, low-risk rollout.

### We just shipped — how do we learn?
Run a retrospective using [Retrospective & Continuous Improvement](octoacme-retrospective-and-continuous-improvement.md) to capture wins and identify improvements.

## Key Artifacts & Templates

Across these guides, you'll find templates for:
- **Project One-pager** — Problem statement, goals, success metrics, stakeholders
- **Backlog Item** — Description, acceptance criteria, priority, estimate
- **Risk Register** — ID, description, impact, likelihood, owner, mitigation
- **Weekly Status** — Progress, next steps, risks, decisions needed
- **Release Notes** — Summary, changes, migration steps, known issues
- **Action Item** — Title, description, owner, due date, success criteria

## Integration with Copilot Spaces

These documents are designed to be added to **Copilot Spaces** for context-aware assistance. When working in a Copilot Space with these docs attached, you can:
- Ask Copilot to explain which phase your project is in and what to do next
- Request a template for a specific artifact (e.g., "give me a risk register template")
- Get role-specific guidance by mentioning your persona
- Ask for examples of how to handle common scenarios (escalations, retrospectives, etc.)

To attach these docs to a Copilot Space:
1. Copy the relevant doc(s) to your project's Copilot context
2. Reference them when asking Copilot questions
3. Use consistent document and section names for better matching
